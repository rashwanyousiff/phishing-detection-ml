\begin{abstract}
Phishing pages are a growing problem and catching them automatically is genuinely useful.
This project explores whether URL and page-content features can reliably separate phishing sites from legitimate ones using three standard classifiers: Logistic Regression, Decision Tree, and Random Forest.
I trained and tested on the PhiUSIIL Phishing URL Dataset (235,795 URLs, 50 features) and found that all three models score essentially perfectly on it, which raised more questions than it answered.
Running the same models on the UCI Phishing Websites dataset revealed the actual differences: LR lands at F1 = 0.917, DT at 0.944, and RF at 0.974.
Looking at which features mattered most, \texttt{URLSimilarityIndex} came out ahead by a large margin, with page-size features like external link count and line-of-code count following behind.

\end{abstract}

\section{Introduction}

Phishing has been around long enough that most people know what it is, yet it keeps working.
Someone builds a page that mimics a bank or an email login screen, gets a user to type in their password, and that is it.
The Anti-Phishing Working Group recorded over 5 million attacks in 2023~\cite{apwg2023}, which is a lot even accounting for some overlap and duplicates in how those numbers get counted.

The appeal of a machine learning solution is obvious: if a model can catch a phishing URL before the user ever sees the page, the whole attack falls apart.
What is less obvious is whether a model trained on one dataset will still work when the next wave of phishing campaigns comes along.
A system that looks great on a benchmark but misses real-world attacks is not worth much.

My starting point for this project was to build a clean, straightforward pipeline and see how three different classifiers compared.
I wanted to understand not just which model scored highest, but whether the features the models leaned on actually made sense as phishing signals.

I did not go in expecting perfect scores.
All three models, including plain Logistic Regression, ended up essentially flawless on PhiUSIIL.
That was genuinely surprising, and it pushed me to take the cross-dataset evaluation more seriously.
The second dataset is where you see the real differences between the models.

\section{Related Work}
\label{sec:related}

The earliest ML work on phishing detection mostly stuck to URL features: how long the URL is, whether the domain looks like an IP address, how many special characters appear, how many subdomains are stacked.
Whittaker et al.~\cite{whittaker2010} demonstrated that even these basic signals, combined with blacklist matching, produced reasonable detection rates.

Things got more interesting when researchers started pulling features from the page content itself, things like whether the page submits forms to external domains, how many iframes or popups it has, and whether it has the kind of social media footprint a real company would have.
Tang and Mahmoud~\cite{tang2021} did a broad survey of this space and found that combining URL and content features consistently outperformed URL-only approaches, with Random Forest-style ensembles tending to come out on top.

One concern that kept coming up in the literature was how poorly these models transferred across datasets.
Zieni et al.~\cite{zieni2023} reviewed over 100 papers and observed that a model trained on data from one time period or collection method often fell apart on data from somewhere else.
That finding directly motivated my decision to evaluate on a second dataset rather than just reporting PhiUSIIL numbers.

PhiUSIIL itself was introduced by Prasad and Chandra~\cite{prasad2024} partly to fill gaps in older benchmarks.
They showed near-perfect performance using gradient-boosted trees, and based on my results, the feature set is rich enough that you get similar numbers from much simpler models too.

\section{Data}
\label{sec:data}

\subsection{PhiUSIIL Phishing URL Dataset}

PhiUSIIL~\cite{uci_phiusiil} has 235,795 labeled URLs spread across 56 columns.
Five of those columns (FILENAME, URL, Domain, TLD, Title) are text-heavy and do not work as direct numeric inputs, so I dropped them and worked with the remaining 50 features.
The class breakdown is 134,850 phishing (57.2\%) and 100,945 legitimate (42.8\%), which is close enough to balanced that I did not bother with oversampling.
No missing values, no duplicate rows.

\begin{figure}[ht]
  \centering
  \includegraphics[width=0.68\linewidth]{fig_class_distribution.png}
  \caption{Class distribution in PhiUSIIL. The 57/43 split is close enough to balanced that no resampling was needed.}
  \label{fig:class_dist}
\end{figure}

The 50 remaining features cover URL structure, page content, and some broader context.
URL-level features capture things like URL length, domain length, TLD type, how many digits or special characters appear, whether the domain is an IP address, and whether the page uses HTTPS.
Content features count images, CSS files, JavaScript files, iframes, popups, outbound links, self-referencing links, and flag things like hidden form fields, password inputs, and external form submission targets.
The contextual group includes line-of-code count, title-to-URL similarity scores, robots.txt presence, redirect counts, and keyword flags for terms like Bank, Pay, and Crypto.

\subsection{UCI Phishing Websites Dataset}

For the second evaluation I used the UCI Phishing Websites dataset~\cite{uci2015}, which comes in ARFF format.
It has 11,055 websites described by 30 integer features, each encoded as either $-1$, $0$, or $1$ depending on whether the feature is phishing-like, uncertain, or legitimate-like.
After flipping the label encoding to match my convention (phishing = 1, legitimate = 0), the breakdown is 4,898 phishing and 6,157 legitimate.
I trained new models on this dataset from scratch using the same 80/20 split, so these numbers are not a transfer test from PhiUSIIL.

\section{Method}
\label{sec:method}

\subsection{Preprocessing}

All preprocessing runs inside scikit-learn \texttt{Pipeline} objects.
This prevents any information from the test set leaking into the scaler during training.
Logistic Regression gets a \texttt{StandardScaler} step because unscaled features will throw off the coefficient magnitudes.
The tree-based models do not care about scale, so their pipelines just hold the model.

PhiUSIIL was split 80/20 with stratification (seed = 42), giving 188,636 training samples and 47,159 test samples.
I checked the class imbalance ratio before training and found it at 0.749, above the 0.6 threshold I set for enabling class weighting, so weights stayed off.

\subsection{Models}

\textbf{Logistic Regression (LR)} models the probability of phishing as a sigmoid applied to a weighted sum of features.
I used L2 regularization with $C = 1.0$ and bumped \texttt{max\_iter} to 1000 to make sure it converged on the larger dataset.
LR is the easiest model to interpret here because each coefficient directly tells you how much a feature pulls toward or away from a phishing prediction.
It also acts as a useful baseline: if LR does badly, the signal in the features is probably not linearly structured.

\textbf{Decision Tree (DT)} builds a binary tree by repeatedly finding the split that most reduces Gini impurity at each node.
I set \texttt{max\_depth = 10} on both datasets to prevent the tree from getting deep enough to memorize individual training examples.

\textbf{Random Forest (RF)} builds 200 trees, each on a bootstrap sample of the data, each choosing features randomly at every split.
The ensemble averaging is what makes it stronger than a single tree.
I also ran a randomized hyperparameter search over 20 configurations with 5-fold CV, which found $n\_\text{estimators} = 300$, $\text{max\_depth} = 30$, and $\text{max\_features} = \sqrt{p}$ as best.
On PhiUSIIL the tuned and default versions perform identically since both hit the ceiling.

\subsection{Evaluation}

I measured accuracy, precision, recall, F1, and AUC on the held-out test split.
I also ran 5-fold stratified cross-validation on the training set for each model.
Cross-validation works by splitting the training data into five equal parts, holding out each one in turn while training on the other four, and averaging the scores.
It gives a less noisy estimate of generalization than a single split.

\subsection{Feature Analysis}

Figure~\ref{fig:rf_imp} shows RF feature importances from mean Gini decrease.
\texttt{URLSimilarityIndex} (0.1974) and \texttt{NoOfExternalRef} (0.1656) are way out in front.
After that comes a cluster of page-volume features: \texttt{LineOfCode} (0.1314), \texttt{NoOfImage} (0.1125), \texttt{NoOfSelfRef} (0.0869), and \texttt{NoOfJS} (0.0754).
Everything else is below 0.034.

\begin{figure}[ht]
  \centering
  \includegraphics[width=0.74\linewidth]{fig_rf_feature_importance.png}
  \caption{RF top-15 feature importances. \texttt{URLSimilarityIndex} leads clearly. Page-volume features fill out the second tier.}
  \label{fig:rf_imp}
\end{figure}

\texttt{URLSimilarityIndex} measures how structurally similar a URL is to known phishing templates.
A page faking a bank login will tend to score high because the URL follows patterns that are well documented in the phishing world.
\texttt{NoOfExternalRef} counts outbound links, and legitimate sites tend to link out a lot while phishing pages are usually sparse.
The page-volume group makes similar sense: a real website takes time to build and has images, stylesheets, scripts, and cross-references, while a hastily deployed phishing form often has almost none of those things.

The LR coefficient plot in Figure~\ref{fig:lr_coef} tells a compatible story.
\texttt{URLSimilarityIndex} has by far the largest positive coefficient at roughly $+7.1$, meaning it pushes very strongly toward a phishing prediction.
\texttt{IsHTTPS} coming in second at $+4.0$ is initially puzzling since HTTPS is supposed to mean a secure connection.
The explanation is that modern phishing kits get free certificates through services like Let's Encrypt, so the data shows phishing pages using HTTPS at a higher rate than some older legitimate pages that still run on plain HTTP.
On the legitimate side, character-ratio features like \texttt{SpacialCharRatioInURL} and \texttt{LetterRatioInURL} are the strongest negative weights, which makes sense because real URLs tend to be mostly alphabetic with conventional punctuation, whereas phishing URLs often look garbled.

\section{Discussion}
\label{sec:discussion}

The PhiUSIIL results are probably better than they should be.
Getting perfect scores from LR on a held-out test set almost always means the dataset has a dominant feature that is doing most of the work, and this case is no exception.
\texttt{URLSimilarityIndex} is likely responsible for most of the separation.
This does not mean the dataset is bad or the result is fake, it just means the benchmark is not a useful place to rank models against each other.
You need harder data for that.

The UCI results explain why RF outperforms the others.
The UCI features are integer codes in $\{-1, 0, 1\}$, which are essentially categorical thresholds.
Tree-based splits are a natural fit for that kind of structure; LR has to turn threshold behavior into a linear approximation, which works but is less efficient.
The performance advantage of RF over DT also makes sense given that UCI has only about 8,800 training samples, a regime where ensemble averaging and feature subsampling provide the most variance reduction.

The broader point is just that one dataset is not enough.
PhiUSIIL would have left the impression that all three models are interchangeable.
Adding UCI immediately showed a spread of about 6 F1 points from bottom to top.
More datasets, ideally from different time periods, would give an even clearer picture.

\section{Limitations}
\label{sec:limitations}

Both datasets are single-time-period snapshots.
Phishing campaigns change fast, and a model that works well on data from one year may struggle with the techniques attackers use the next year.
Ideally you would train on earlier data and test on later data, but neither of these datasets is set up for that.

A practical complication is that many of the most useful features require actually loading the page.
Things like JavaScript file counts, external link counts, and line-of-code totals all depend on fetching and parsing HTML.
That adds latency and makes the approach less viable as a pre-click URL filter.

On PhiUSIIL, running a hyperparameter search found different settings but produced the same score, which is what you would expect when the baseline is already at 1.0.
On UCI I only tuned DT depth and RF tree count; LR's regularization strength and other settings were left at defaults.

Two datasets are better than one, but they are still both curated academic benchmarks.
Results on them may not reflect how the models would hold up against a real phishing campaign collected in the wild.

\section{Conclusion}
\label{sec:conclusion}

I trained Logistic Regression, Decision Tree, and Random Forest on two phishing datasets.
On PhiUSIIL, all three score near perfectly, which turned out to say more about the dataset than the models.
On UCI Phishing Websites, RF finishes at F1 = 0.974, DT at 0.944, and LR at 0.917.

Feature importance and LR coefficients both point to \texttt{URLSimilarityIndex} as the dominant signal, with page-volume features filling in behind it.
LR's coefficient plot also highlights URL character ratios as some of the strongest signals on the legitimate side, which lines up with how real URLs tend to look compared to phishing ones.

The main thing I would take away from this project is that a benchmark where a linear model scores perfectly is probably too easy to tell you much.
Running on a second, harder dataset is what actually revealed how the three classifiers compare.
